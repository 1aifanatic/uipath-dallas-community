Core idea

Use the lightest build path that can survive production.

Great UiPath architecture is not about choosing the newest option. It is about matching the work, team, risk, and operating model.

Work shapeIs it deterministic, judgment-heavy, or multi-actor?
Builder profileWho owns it after the demo: citizen, automation team, or engineers?
Control needsWhat audit, testing, approval, and policy gates are required?
Operating modelWho monitors, changes, and improves it after launch?

Quick Selector Matrix

Use this as the first conversation with sponsors and builders.

Path Use when Watch out for Production focus
Studio Clear rules, repeatable inputs, system execution. Do not force AI where validation and queues solve it. Queues, assets, logging, exceptions.
Agent Builder Reasoning with bounded tools and human fallback. Keep tools approved and scope narrow. Guardrails, evals, HITL handoff.
Coded Agents Custom code, SDKs, libraries, tests, repos. Do not rebuild platform services in code. Repo ownership, CI, observability.
Maestro Multiple roles, long-running state, pause/resume. Do not use it only to draw a workflow. State, SLA, audit, operations.

Build Path 1: Studio

For deterministic automation where robots execute repeatable work reliably.

Use Studio when

  • The steps are known and repeatable.
  • Inputs are structured or can be validated.
  • Queue-driven work improves reliability.
Best matched to clear rules

Design pattern

Trigger, validate input, create queue item, execute transaction, update system, write evidence.

Make failure observable

Demo cue

Show a deterministic intake robot with early validation, queue status, and traceable output.

Automation with discipline
Prev 8 / 22 Next

Build Path 2: Agent Builder

For reasoning work that needs tools, context, guardrails, and human safety.

1PlanGoal, instructions, knowledge.
2GroundApproved tools and references.
3ActDraft, classify, enrich, recommend.
4EvaluateTrace, thresholds, feedback.
5EscalateHuman-in-the-loop when needed.

Choose Agent Builder when the experience should stay approachable but the work needs judgment.

Prev 9 / 22 Next

Build Path 3: Coded Agents

For engineering-led agents that need source control, SDKs, tests, and custom behavior.

When code earns its place

  • Custom frameworks or libraries.
  • Shared services and APIs.
  • Rigorous tests and CI/CD.
  • Repository ownership.
  • Developer standards after launch.

Good coded agents feel like software projects that happen to run inside the automation platform.

Production Readiness: Seven Gates

A build path only counts when it can run, fail, recover, and improve.

1Data and context
2RBAC and secrets
3Tool boundaries
4Evaluations and smoke tests
5Human fallback
6Logs, traces, evidence
7Owner and support model

For agents, run smoke tests from workflows, inspect traces, verify readiness, and design human escalation before launch.

Build-Path Clinic

Use these scenarios to make the decision framework tangible.

Vendor onboarding

Documents arrive inconsistently. Policy exceptions need review. System updates are repeatable.

Likely: Maestro plus Agent Builder plus Studio

Daily report automation

Predictable data pulls, transform, validate, and email distribution.

Likely: Studio workflow

Custom research agent

Needs Python libraries, repo ownership, and evaluation harness.

Likely: Coded Agent

Service desk triage

Classify cases, suggest drafts, verify tool-approved facts, escalate low confidence.

Likely: Agent Builder
